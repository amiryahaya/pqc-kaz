# 09. SSDID Cryptographic Specification

## 1. Overview

This document defines the cryptographic operations that all SSDID implementations (Elixir backend, Android SDK, iOS SDK) must follow to achieve interoperability. Any implementation that conforms to this specification will produce signatures verifiable by any other conforming implementation.

### 2.2 Key Sizes

**Classical:**

| Algorithm | Secret Key | Public Key | Signature |
|-----------|-----------|------------|-----------|
| Ed25519 | 32 bytes | 32 bytes | 64 bytes |
| ECDSA P-256 | 32 bytes | 65 bytes (uncompressed) | ~70 bytes (DER) |
| ECDSA P-384 | 48 bytes | 97 bytes (uncompressed) | ~104 bytes (DER) |

**PQC (ML-DSA) — NIST FIPS 204:**

| Algorithm | Public Key | Signature | Security Level |
|-----------|-----------|-----------|----------------|
| ML-DSA-44 | ~1,312 bytes | ~2,420 bytes | 128-bit |
| ML-DSA-65 | ~1,952 bytes | ~3,293 bytes | 192-bit |
| ML-DSA-87 | ~2,592 bytes | ~4,595 bytes | 256-bit |

**PQC (SLH-DSA) — NIST FIPS 205:**

Sizes vary by variant. "s" = small signatures, "f" = fast signing.

| Variant | Public Key | Signature (s) | Signature (f) |
|---------|-----------|---------------|---------------|
| 128 | 32 bytes | ~7,856 bytes | ~17,088 bytes |
| 192 | 48 bytes | ~16,224 bytes | ~35,664 bytes |
| 256 | 64 bytes | ~29,792 bytes | ~49,856 bytes |

**KAZ-Sign (3 security levels):**

| Level | Secret Key | Public Key | Signature Overhead | Total Signature |
|-------|-----------|------------|-------------------|-----------------|
| 128 | 32 bytes | 54 bytes | 162 bytes | 162 + msg_len |
| 192 | 50 bytes | 88 bytes | 264 bytes | 264 + msg_len |
| 256 | 64 bytes | 118 bytes | 356 bytes | 356 + msg_len |

Note: KAZ-Sign uses message-recovery natively. The extended C library (doc 10) adds a detached signing mode for SSDID protocol compatibility.

### 2.3 Key Encoding in DID Documents

**Classical keys (Ed25519, ECDSA):**
- Public key bytes encoded as multibase base64url-nopad
- Format in DID Document: `"publicKeyMultibase": "u<base64url_encoded_bytes>"`
- Ed25519: 32 raw bytes -> ~43 chars after encoding
- ECDSA: uncompressed point (0x04 || x || y) -> 65 or 97 bytes

**PQC keys (ML-DSA, SLH-DSA):**
- Public key is the X.509 SubjectPublicKeyInfo DER encoding
- Encoded as multibase base64url-nopad
- These are larger: ~1.3KB to ~2.6KB for ML-DSA

**KAZ-Sign keys:**
- Public key is the raw binary (big-endian, fixed-length per level)
- Encoded as multibase base64url-nopad
- 54/88/118 bytes depending on security level

## 3. Multibase Encoding

All public keys and signatures in SSDID use **multibase base64url-nopad** encoding.

### 3.1 Encoding

```
multibase_encode(bytes) = "u" + base64url_encode_nopad(bytes)
```

Where `base64url_encode_nopad` uses the URL-safe alphabet (`A-Z`, `a-z`, `0-9`, `-`, `_`) with **no padding characters** (`=`).

The `u` prefix identifies the encoding as base64url per the multibase specification.

### 3.2 Decoding

```
multibase_decode(encoded):
  assert encoded[0] == "u"
  return base64url_decode(encoded[1:])
```

### 3.3 Examples

```
Raw bytes (hex):  48 65 6c 6c 6f
Base64url-nopad:  SGVsbG8
Multibase:        uSGVsbG8
```

### 3.4 Platform Implementation

**Kotlin/Android:**
```kotlin
fun encodeMultibase(bytes: ByteArray): String =
    "u" + Base64.encodeToString(bytes, Base64.URL_SAFE or Base64.NO_PADDING or Base64.NO_WRAP)

fun decodeMultibase(encoded: String): ByteArray {
    require(encoded.startsWith("u"))
    return Base64.decode(encoded.substring(1), Base64.URL_SAFE or Base64.NO_PADDING)
}
```

**Swift/iOS:**
```swift
func encodeMultibase(_ bytes: Data) -> String {
    "u" + bytes.base64URLEncodedString()  // no padding
}

func decodeMultibase(_ encoded: String) -> Data? {
    guard encoded.hasPrefix("u") else { return nil }
    return Data(base64URLEncoded: String(encoded.dropFirst()))
}
```

## 4. Canonical JSON

Deterministic JSON serialization ensures that identical data structures produce identical byte sequences across all platforms. This is critical for signatures — the signing payload includes hashes of JSON-serialized data.

### 4.1 Rules

1. **Sort map/object keys alphabetically** (lexicographic, Unicode code point order)
2. **No whitespace** between tokens (no spaces after `:` or `,`)
3. **Recurse** into nested maps and arrays
4. **Standard JSON encoding** for strings (escape `"`, `\`, control characters)
5. **No trailing commas**
6. **Numbers**: integers as integers (no `.0`), floats with minimal representation

### 4.2 Examples

**Input (unsorted):**
```json
{"b": 2, "a": 1, "c": {"z": 26, "a": 1}}
```

**Canonical output:**
```json
{"a":1,"b":2,"c":{"a":1,"z":26}}
```

**Array (order preserved, not sorted):**
```json
{"items": [3, 1, 2]}
```

**Canonical output:**
```json
{"items":[3,1,2]}
```

### 4.3 Verification

To verify your implementation matches the Elixir backend:

```
Input:  {"verificationMethod":"did:ssdid:abc#key-1","type":"Ed25519Signature2020","proofPurpose":"assertionMethod","created":"2024-01-01T00:00:00Z"}

Canonical: {"created":"2024-01-01T00:00:00Z","proofPurpose":"assertionMethod","type":"Ed25519Signature2020","verificationMethod":"did:ssdid:abc#key-1"}
```

Keys sorted: `created` < `proofPurpose` < `type` < `verificationMethod`.

## 5. Proof Construction

### 5.1 Signing Payload

The signing payload for a proof is constructed as follows:

```
signing_payload = SHA3-256(canonical_json(proof_options)) || SHA3-256(canonical_json(document_without_proof))
```

Where `||` denotes byte concatenation.

**Step by step:**

1. Remove `"proof"` key from the document (if present) -> `doc_clean`
2. Build proof options map with required fields (see below) -> `proof_options`
3. Remove `"proofValue"` from proof options (if present) -> `options_clean`
4. `options_hash = SHA3-256(canonical_json(options_clean))` — 32 bytes
5. `doc_hash = SHA3-256(canonical_json(doc_clean))` — 32 bytes
6. `signing_payload = options_hash || doc_hash` — 64 bytes total
7. `signature = sign(signing_payload, secret_key, algorithm)`
8. `proof_options["proofValue"] = multibase_encode(signature)`

**Important:** The order is `options_hash` first, then `doc_hash`. This follows the W3C Data Integrity specification.

### 5.2 Proof Options

Required fields:

```json
{
  "type": "<W3C_Proof_Type>",
  "created": "<ISO8601_UTC>",
  "verificationMethod": "<did>#<key_fragment>",
  "proofPurpose": "<purpose>"
}
```

Optional fields (included when present):

```json
{
  "challenge": "<challenge_string>",
  "domain": "<domain_string>"
}
```

### 5.3 Proof Verification

To verify a proof:

1. Extract `proofValue` from the proof -> `encoded_signature`
2. `signature = multibase_decode(encoded_signature)`
3. Build `proof_options` from the proof, excluding `proofValue`
4. `signing_payload = SHA3-256(canonical_json(proof_options)) || SHA3-256(canonical_json(document_without_proof))`
5. Resolve the DID Document for the `verificationMethod` DID
6. Find the verification method matching the key ID
7. Extract the public key: `public_key = multibase_decode(publicKeyMultibase)`
8. Determine algorithm from the verification method `type` (see W3C type mappings)
9. `result = verify(signature, signing_payload, public_key, algorithm)`

### 5.4 Proof Purpose Validation

The proof's `proofPurpose` must match a verification relationship in the DID Document:

| Proof Purpose | DID Document Field |
|--------------|-------------------|
| `authentication` | `authentication` |
| `assertionMethod` | `assertionMethod` |
| `capabilityInvocation` | `capabilityInvocation` |
| `capabilityDelegation` | `capabilityDelegation` |
| `keyAgreement` | `keyAgreement` |

The key ID in `verificationMethod` must appear in the corresponding array.

## 6. DID Generation

### 6.1 Format

```
did:ssdid:<method_specific_id>
```

### 6.2 Method-Specific ID Generation

1. Generate 128 bits (16 bytes) of cryptographically secure random data
2. Encode using Base64url with no padding
3. Result: 22 characters

```
random_bytes = CSPRNG(16)
method_specific_id = base64url_encode_nopad(random_bytes)
did = "did:ssdid:" + method_specific_id
```

### 6.3 Example

```
Random (hex): a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6
Base64url:    obLD1OX2p7jJ0OHyo7TF1g
DID:          did:ssdid:obLD1OX2p7jJ0OHyo7TF1g
```

## 7. Challenge-Response Signing

### 7.1 Simple Challenge (Registration)

During registration, the client signs the raw challenge bytes:

```
signed_challenge = sign(challenge_bytes, secret_key, algorithm)
```

The server verifies by resolving the client's DID Document and checking the signature.

### 7.2 Transaction-Bound Challenge

For transactions, the signature binds the challenge to the transaction body:

```
tx_json = canonical_json(transaction)
tx_hash = SHA3-256(tx_json)
tx_hash_encoded = base64url_encode_nopad(tx_hash)
signing_payload = challenge_string + tx_hash_encoded
signed_challenge = sign(signing_payload_bytes, secret_key, algorithm)
```

Note: `challenge_string` is the UTF-8 string as received from the server. `tx_hash_encoded` is the base64url-nopad string of the hash. The two strings are concatenated and then the resulting UTF-8 bytes are signed.

This prevents:
- **Replay attacks**: Challenge is single-use, consumed after verification
- **Transaction substitution**: Signature is bound to the exact transaction hash
- **Hash collision attacks**: SHA3-256 provides 128-bit collision resistance

## 8. SHA3-256

All hashing in SSDID uses **SHA3-256** (FIPS 202), producing a 32-byte (256-bit) digest.

Used in:
- Proof signing payload (hash of document, hash of proof options)
- Transaction binding (hash of transaction body)

**Not SHA-256.** SSDID uses SHA3-256 (Keccak-based), not SHA-256 (SHA-2 family).

### 8.1 Platform Implementation

**Kotlin/Android:**
```kotlin
import java.security.MessageDigest

fun sha3_256(data: ByteArray): ByteArray =
    MessageDigest.getInstance("SHA3-256").digest(data)
```

**Swift/iOS:**
```swift
import CryptoKit

func sha3_256(_ data: Data) -> Data {
    // CryptoKit does not include SHA3. Use OpenSSL or a Swift SHA3 library.
    // Example with a SHA3 library:
    Data(SHA3.hash256(data))
}
```

Note: iOS CryptoKit (as of iOS 18) does not include SHA3-256. Use CommonCrypto with OpenSSL, or a pure Swift SHA3 implementation. The PQC-KAZ/SIGN C library (doc 10) can also expose SHA3-256 via OpenSSL for iOS use.

## 9. Signing and Verification

### 9.1 Classical Algorithms

**Ed25519:**
- Sign: `Ed25519_Sign(message, secret_key)` -> 64-byte signature
- Verify: `Ed25519_Verify(signature, message, public_key)` -> boolean
- No pre-hashing — Ed25519 handles hashing internally (SHA-512)

**ECDSA P-256 / P-384:**
- Sign: `ECDSA_Sign(SHA3-256(message), secret_key)` -> DER-encoded signature
- Verify: `ECDSA_Verify(signature, SHA3-256(message), public_key)` -> boolean
- Note: The SSDID proof construction already hashes the document; ECDSA signs the resulting 64-byte payload directly

### 9.2 PQC Algorithms (ML-DSA, SLH-DSA)

**Signing flow (matches ap_java_crypto):**
1. Compute `digest = SHA3-256(data)` on the application side
2. Pass `digest` to the PQC signer
3. The PQC algorithm signs the digest bytes directly
4. Return raw signature bytes

**Verification flow:**
1. Compute `digest = SHA3-256(data)` on the application side
2. Pass `(digest, signature, public_key)` to the PQC verifier
3. Return boolean

**Key encoding for ML-DSA and SLH-DSA:**
- Private keys: PKCS8 DER encoding
- Public keys: X.509 SubjectPublicKeyInfo DER encoding
- In DID Documents: `publicKeyMultibase = "u" + base64url_nopad(der_bytes)`

### 9.3 KAZ-Sign

**Signing flow (detached mode, per extended C library — doc 10):**
1. Compute `digest = SHA3-256(data)` inside the C library
2. Sign the digest using KAZ-Sign algorithm
3. Return detached signature bytes (not message-recovery format)

**Verification flow:**
1. Compute `digest = SHA3-256(data)` inside the C library
2. Verify `(signature, digest, public_key)` -> boolean

**Key encoding for KAZ-Sign:**
- Raw binary format (big-endian, fixed-length per security level)
- In DID Documents: `publicKeyMultibase = "u" + base64url_nopad(raw_pubkey_bytes)`

## 11. Interoperability Test Vectors

Implementers should verify their implementations produce identical outputs for these test cases. Exact test vectors with known keys and expected signatures will be published in a separate test vector document. The critical invariants to test:

1. **Canonical JSON**: Given the same map, all implementations produce byte-identical JSON
2. **Multibase**: Given the same bytes, all implementations produce the same encoded string
3. **SHA3-256**: Given the same input, all implementations produce the same 32-byte hash
4. **Proof signing payload**: Given the same document and proof options, the 64-byte signing payload is identical
5. **Signature verification**: A signature produced by any implementation verifies on all other implementations
