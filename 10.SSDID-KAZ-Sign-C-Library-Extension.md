# 10. SSDID KAZ-Sign C Library Extension Specification

## 1. Overview

This document specifies extensions to the existing **PQC-KAZ/SIGN** C library to support the full feature set required by the SSDID protocol. The goal is to bring the C library to feature parity with `ap_java_crypto`'s KAZ-Sign capabilities, scoped to KAZ-Sign only (no ML-DSA or SLH-DSA).

The extended library will be used by:
- Android SDK (via existing JNI/Kotlin bindings)
- iOS SDK (via existing Swift bindings)
- Elixir backend (via existing NIF bindings, as an alternative to the JRuby/Java path)

## 2. Current vs Target Feature Matrix

| Feature | Current (PQC-KAZ/SIGN v2.1) | Target (Extended) |
|---------|------------------------------|-------------------|
| Key generation | 3 levels (128/192/256) | Same |
| Message-recovery signing | Yes | Yes (unchanged) |
| **Detached signing** | No | **Add** |
| **SHA3-256 pre-hashing** | No (SHA-256/384/512) | **Add** |
| **DER key encoding** | No (raw binary) | **Add** |
| **X.509 CSR generation** | No | **Add** |
| **X.509 certificate issuance** | No | **Add** |
| **X.509 certificate verification** | No | **Add** |
| **PKCS12 keystore** | No | **Add** |
| HKDF (RFC 5869) | Yes | Same |
| Constant-time operations | Yes | Same |
| Android bindings (JNI/Kotlin) | Yes | Extend |
| iOS bindings (Swift) | Yes | Extend |
| Elixir bindings (NIF) | Yes | Extend |

## 3. New C API

All new functions follow the existing naming convention (`kaz_sign_*`) and error code pattern.

### 3.1 Detached Signing

Signs a pre-computed hash (or raw data with internal SHA3-256), returning only the signature without the message embedded.

```c
/**
 * Sign data in detached mode (signature does not contain the message).
 * Internally computes SHA3-256(data) before signing.
 *
 * @param level     Security level (128, 192, or 256)
 * @param sig       Output buffer for signature (kaz_sign_detached_sig_bytes(level))
 * @param siglen    Output: actual signature length
 * @param data      Input data to sign
 * @param datalen   Length of input data
 * @param sk        Secret key (kaz_sign_secret_key_bytes(level))
 * @return          KAZ_SIGN_SUCCESS or error code
 */
int kaz_sign_detached_ex(
    kaz_sign_level_t level,
    unsigned char *sig,
    unsigned long long *siglen,
    const unsigned char *data,
    unsigned long long datalen,
    const unsigned char *sk
);

/**
 * Verify a detached signature.
 * Internally computes SHA3-256(data) before verification.
 *
 * @param level     Security level
 * @param data      Original data that was signed
 * @param datalen   Length of original data
 * @param sig       Signature to verify
 * @param siglen    Signature length
 * @param pk        Public key
 * @return          KAZ_SIGN_SUCCESS if valid, KAZ_SIGN_ERROR_VERIFY if invalid
 */
int kaz_sign_verify_detached_ex(
    kaz_sign_level_t level,
    const unsigned char *data,
    unsigned long long datalen,
    const unsigned char *sig,
    unsigned long long siglen,
    const unsigned char *pk
);

/**
 * Sign a pre-computed digest in detached mode (caller already hashed).
 * Use this when the caller has already computed SHA3-256.
 *
 * @param level     Security level
 * @param sig       Output buffer for signature
 * @param siglen    Output: actual signature length
 * @param digest    Pre-computed SHA3-256 digest (32 bytes)
 * @param sk        Secret key
 * @return          KAZ_SIGN_SUCCESS or error code
 */
int kaz_sign_detached_prehashed_ex(
    kaz_sign_level_t level,
    unsigned char *sig,
    unsigned long long *siglen,
    const unsigned char *digest,
    const unsigned char *sk
);

/**
 * Verify a detached signature against a pre-computed digest.
 *
 * @param level     Security level
 * @param digest    Pre-computed SHA3-256 digest (32 bytes)
 * @param sig       Signature to verify
 * @param siglen    Signature length
 * @param pk        Public key
 * @return          KAZ_SIGN_SUCCESS if valid, KAZ_SIGN_ERROR_VERIFY if invalid
 */
int kaz_sign_verify_detached_prehashed_ex(
    kaz_sign_level_t level,
    const unsigned char *digest,
    const unsigned char *sig,
    unsigned long long siglen,
    const unsigned char *pk
);
```

**Detached signature format:**

The detached signature is the S1 || S2 || S3 components only (no message). The message is provided separately during verification.

| Level | Detached Signature Size |
|-------|------------------------|
| 128 | 162 bytes |
| 192 | 264 bytes |
| 256 | 356 bytes |

### 3.2 SHA3-256

Expose SHA3-256 hashing via OpenSSL EVP for use by the mobile SDKs (particularly iOS, which lacks native SHA3-256).

```c
/**
 * Compute SHA3-256 digest.
 *
 * @param data      Input data
 * @param datalen   Length of input data
 * @param digest    Output buffer (32 bytes)
 * @return          KAZ_SIGN_SUCCESS or error code
 */
int kaz_sha3_256(
    const unsigned char *data,
    unsigned long long datalen,
    unsigned char *digest
);

/**
 * Incremental SHA3-256 hashing.
 */
typedef struct kaz_sha3_ctx kaz_sha3_ctx_t;

int kaz_sha3_256_init(kaz_sha3_ctx_t **ctx);
int kaz_sha3_256_update(kaz_sha3_ctx_t *ctx, const unsigned char *data, unsigned long long datalen);
int kaz_sha3_256_final(kaz_sha3_ctx_t *ctx, unsigned char *digest);
void kaz_sha3_256_free(kaz_sha3_ctx_t *ctx);
```

    kaz_sign_level_t level,
    const unsigned char *sk,
    unsigned char *der,
    unsigned long long *derlen

**DER format notes:**

The DER encoding must be compatible with the Java `KAZSIGNPublicKey.from_encoded()` and `KAZSIGNPrivateKey.from_encoded()` methods used by `ap_java_crypto`. This requires matching the OID and structure used by `kaz-pqc-jcajce-0.0.2.jar`.

## 5. Implementation Notes

### 5.1 Detached Signing Internals

The detached sign operation:

1. Compute `digest = SHA3-256(data)` using OpenSSL EVP
2. Call the existing KAZ-Sign core with the digest as the "message"
3. Extract only the S1 || S2 || S3 components from the signature (discard the embedded message portion)
4. Return S1 || S2 || S3 as the detached signature

The detached verify operation:

1. Compute `digest = SHA3-256(data)` using OpenSSL EVP
2. Reconstruct the full message-recovery signature: S1 || S2 || S3 || digest
3. Call the existing KAZ-Sign verify core
4. If verification succeeds and the recovered message matches the digest, return success

### 5.2 DER Encoding Compatibility

The DER encoding must produce output that `kaz-pqc-jcajce-0.0.2.jar` can parse via `KAZSIGNPublicKey.from_encoded()` / `KAZSIGNPrivateKey.from_encoded()`. Steps:

1. Determine the OID used by the Java library for each security level
2. Construct SubjectPublicKeyInfo / PrivateKeyInfo ASN.1 structures matching the Java format
3. Verify round-trip: C encode -> Java decode -> Java encode -> C decode

### 5.3 X.509 Implementation

Use OpenSSL's X.509 API (`X509_*`, `X509_REQ_*`) with a custom `EVP_PKEY` type for KAZ-Sign, or construct the ASN.1 structures manually using OpenSSL's ASN.1 primitives:

1. Build the TBS (To Be Signed) certificate structure
2. Hash the TBS with SHA3-256
3. Sign the hash using `kaz_sign_detached_prehashed_ex()`
4. Assemble the final X.509 certificate with the signature

### 5.4 OpenSSL Dependencies

New OpenSSL APIs used by the extensions:

| Feature | OpenSSL API |
|---------|-------------|
| SHA3-256 | `EVP_sha3_256()`, `EVP_DigestInit_ex()`, `EVP_DigestUpdate()`, `EVP_DigestFinal_ex()` |
| DER encoding | `i2d_PUBKEY()`, `d2i_PUBKEY()`, `i2d_PKCS8_PRIV_KEY_INFO()`, `d2i_PKCS8_PRIV_KEY_INFO()` |
| X.509 | `X509_new()`, `X509_REQ_new()`, `X509_sign()`, `X509_verify()` |
| PKCS12 | `PKCS12_create()`, `PKCS12_parse()` |

Minimum OpenSSL version: **3.0** (required for SHA3-256 EVP support).

## 7. Testing Requirements

### 7.1 New Unit Tests

| Test File | Coverage |
|-----------|----------|
| `test_detached.c` | Detached sign/verify round-trip, prehashed mode, wrong key rejection, empty data |
| `test_sha3.c` | SHA3-256 against known test vectors (NIST FIPS 202), incremental hashing |
| `test_der.c` | DER encode/decode round-trip, cross-compatibility with Java `from_encoded()` |
| `test_x509.c` | CSR generation/verification, self-signed cert, CA-signed cert, cert chain |
| `test_p12.c` | P12 create/load round-trip, password protection, chain preservation |

### 7.2 Cross-Platform Interoperability Tests

Critical: signatures and keys produced by the C library must be verifiable by `ap_java_crypto`, and vice versa.

| Test | Description |
|------|-------------|
| C sign -> Java verify | Sign with `kaz_sign_detached_ex()`, verify with `ApJavaCrypto.verify()` |
| Java sign -> C verify | Sign with `ApJavaCrypto.sign()`, verify with `kaz_sign_verify_detached_ex()` |
| C DER -> Java decode | `kaz_sign_pubkey_to_der()` output parsed by `KAZSIGNPublicKey.from_encoded()` |
| Java DER -> C decode | `KAZSIGNPublicKey.getEncoded()` parsed by `kaz_sign_pubkey_from_der()` |
| C P12 -> Java load | `kaz_sign_create_p12()` loaded by `ApJavaCrypto.load_p12()` |
| Java P12 -> C load | `ApJavaCrypto.generate_p12()` loaded by `kaz_sign_load_p12()` |

### 7.3 Constant-Time Verification

All new code paths must pass the existing DudeCT timing analysis. In particular:
- `kaz_sign_detached_ex()` must not leak timing based on message content
- DER encoding must not introduce timing side channels for secret keys

## 8. Migration Path

### 8.1 Phase 1: Detached Signing + SHA3-256

Priority features for SSDID protocol compatibility:
- `kaz_sign_detached_ex()` / `kaz_sign_verify_detached_ex()`
- `kaz_sha3_256()`
- Binding updates for Android and iOS

This phase alone enables mobile SDKs to participate in the SSDID protocol.

### 8.2 Phase 2: DER Encoding

- `kaz_sign_pubkey_to_der()` / `kaz_sign_pubkey_from_der()`
- `kaz_sign_privkey_to_der()` / `kaz_sign_privkey_from_der()`
- Cross-compatibility testing with Java `from_encoded()`

### 8.3 Phase 3: X.509 + PKCS12

- CSR generation
- Certificate issuance and verification
- PKCS12 keystore
- Full cross-platform testing

## 9. Security Considerations

- All new code must maintain the constant-time guarantees of the existing library
- Secret keys must be securely zeroed after use (`kaz_secure_zero()`)
- PKCS12 passwords must not be stored in memory longer than necessary
- DER encoding of private keys must use secure memory allocation
- X.509 certificate parsing must validate all fields to prevent malformed input attacks
- OpenSSL error states must be properly checked and cleared
